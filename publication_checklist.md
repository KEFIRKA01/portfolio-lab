# Publication Checklist

## Порядок публикации

1. Проверить и локально прогнать каждый проект из `projects/`.
2. Разнести проекты в отдельные GitHub-репозитории.
3. Поднять `live demo`:
   - `Render/Railway` для API и Python-проектов;
   - `Vercel/Netlify` для фронтенд-проектов;
   - `GitHub Pages` для документации и case pages;
   - локальный демо-стенд WordPress для WooCommerce-плагина.
4. Снять реальные скриншоты экранов по shot-list.
5. Записать короткие GIF.
6. Опубликовать кейсы на биржах.

## Чек-лист одного проекта

- есть `README.md`;
- есть `CASE.md`;
- есть `ARCHITECTURE.md`;
- есть код, который можно показать или развернуть;
- есть seed/demo data;
- есть `.env.example`, если нужен конфиг;
- есть shot-list для реальных скриншотов;
- есть краткий one-page summary для биржи;
- есть финальная обложка.

## Правило честности

Публикация идёт только с формулировками, которые можно защитить фактами:

- "демонстрационный публичный проект" — можно;
- "создан с нуля и опубликован в GitHub" — можно после публикации;
- "решает типовую задачу бизнеса" — можно;
- "внедрено у клиента" — нельзя, если клиента не было;
- "увеличили продажи" — нельзя, если это не подтверждено.
